# Skill Wiki — 市场研究

对八个现有技能/模块/包市场的调查，用于指导 Skill Wiki 网站的信息架构。每个市场记录了 URL、主要 UX 模式、三个值得借鉴的点，以及一个要避免的问题。页面通过 WebFetch 检视，部分受限时辅以网络搜索。

---

## 1. Anthropic Skills Hub
**URL：** https://github.com/anthropics/skills
**UX 模式：** 浏览优先，GitHub 原生。README 即索引；顶层 `skills/` 目录包含自包含文件夹，每个文件夹有一个带 YAML 元数据（`name`、`description`）的 `SKILL.md`。没有任何搜索或筛选——发现靠阅读 README 和点击文件夹。分类（"Creative & Design"、"Development & Technical"等）是平铺标题。

**值得借鉴：**（1）自包含文件夹惯例——每个技能是一个目录，不是数据库中的一行。（2）仅必要的极简 YAML 元数据；其余全是内容。（3）激进的极简主义——无安装计数、无星标、无排名。信号即策展者（Anthropic）。

**要避免：** 完全没有搜索。技能超过约 50 个后，滚动 README 就不再管用了。我们至少需要筛选，即使没有全文搜索。

## 2. npm 注册中心 Web UI
**URL：** https://www.npmjs.com（页面对机器人返回 403；辅以搜索）
**UX 模式：** 搜索优先。页眉被搜索框主导；详情页显示安装命令、README，侧边栏含版本/依赖/被依赖/周下载量/维护者/关键词。

**值得借鉴：**（1）每个详情页顶部的醒目一键复制安装命令（`prime install @recipes/method-pan-sauce`）。（2）侧边栏元数据面板作为稳定的可用性模式——版本、类型、领域、出边数、文件大小。（3）版本标签页——显示每个已发布版本及日期。

**要避免：** 臃肿的 Chrome 和慢速搜索。2026 年社区对 npmjs.com 的反馈催生了替代浏览器（`npmx`），正是因为官方 UI 显得沉重。我们的页面应以文字为主、加载快速。

---

## 3. Hugging Face Hub
**URL：** https://huggingface.co/models
**UX 模式：** 筛选优先，侧边栏驱动。左侧栏提供约 50 个任务筛选、约 40 个库筛选、参数大小滑块、语言、许可证、推理提供商。卡片显示任务徽章、参数徽章、下载量、点赞数、更新时间。

**值得借鉴：**（1）多正交轴筛选侧边栏（任务/库/语言）——对我们来说：类型/领域/边动词。（2）热门排序 + 最近更新排序作为默认标签页。（3）卡片上的定量参与度信号——我们可能没有下载量，但有边出度和原子质量评分。

**要避免：** 筛选碎片化。五十个复选框会引发选择恐惧症。将筛选器分组在可折叠标题下；最常用的三个显示在顶部。

## 4. crates.io
**URL：** https://crates.io
**UX 模式：** 混合型。首页并排显示"最多下载"、"刚刚发布"、"最近下载最多"三列。详情页显示 README、版本下拉菜单、Cargo.toml 式安装片段、依赖项、所有者、docs.rs 链接。

**值得借鉴：**（1）首页三列精选列表（我们会用"最多边数"、"最新添加"、"高质量"作为三个角度）。（2）从每个 crate 直接链接到其渲染文档（`docs.rs/<crate>`）；我们应将每个原子链接到其 `chunks/full.md` 和渲染后的图。（3）分类页面（`/categories`）将分类法作为头等名词。

**要避免：** 仅按下载量排名。一个新但高质量的技能不应被淹没在有五年积累下载量的技能之下。我们的排名函数应权衡边中心性和质量，而非仅靠流行度。

## 5. PyPI
**URL：** https://pypi.org
**UX 模式：** 基础基线型。搜索优先首页附原始统计数字（"804,996 个项目，860 万个版本"）。项目页面显示描述、版本历史、项目链接、分类器。没有搜索以外的按类别浏览。

**值得借鉴：**（1）项目链接侧边栏（主页/文档/源码/问题），作为稳定的可用性模式。（2）版本历史作为时间顺序列表——传达维护活跃程度。（3）对规模的诚实；数字作为社会证明而无需游戏化。

**要避免：** 没有浏览模式。PyPI 假定你知道包名。对于发现型网站这是死路——我们从第一天起就会有浏览页面。

---

## 6. VS Code Marketplace
**URL：** https://marketplace.visualstudio.com
**UX 模式：** 策展主导的分类页面加卡片网格。每个扩展卡片显示图标、名称、发布者、安装量、五星评分。详情页有标签页：概述、变更日志、问答、评分与评论，以及"资源"侧边栏。

**值得借鉴：**（1）标签式详情页（我们会用 Summary / Core / Full / Edges / Provenance 代替营销标签）。（2）每张卡片上可见发布者/来源——我们会显示领域前缀 `@recipes` / `@team` 等。（3）从 manifest 自动提取的"功能贡献"——对我们来说：按动词列出的出边，在构建时计算。

**要避免：** 星级评分。对知识产物的五星评论毫无意义（"我给 Maillard 反应打 4 星"）。用客观信号（边出度、质量评分）代替。

---

## 7. Cursor Directory
**URL：** https://cursor.directory/
**UX 模式：** 浏览优先，分类主导。本站是规范要求我们镜像的视觉参考——英雄横幅、分类网格、密集卡片布局。每张卡片有标题、简短描述、标签、复制规则按钮。移动端折叠为单列。

**值得借鉴：**（1）每张卡片上的复制按钮（`prime install ...` 随时可粘贴）。（2）分类即页面（`/category/cooking`）与我们想要的领域结构一致。（3）安静的单色设计——内容承载信号，无渐变英雄区或营销噱头。

**要避免：** 单薄的详情页。Cursor 的单个规则页面基本上只是规则文本加复制按钮。我们的原子有边和投影层级——我们应该暴露这些，而非埋藏。

## 8. awesome-claude-skills（GitHub 话题）
**URL：** https://github.com/topics/claude-skills
**UX 模式：** 草根聚合型。约 3,000 个仓库按星标排序。常见 README 格式：徽章、简短说明、目录、分类列表、贡献说明。发现靠语言筛选和星标排名。

**值得借鉴：**（1）星标数作为信号——对我们来说，类比是社区认可（接受的 PR、引用的语料库）。（2）Markdown 中的分类列表——易于渲染、易于 fork、无需基础设施。（3）交叉链接到其他生态系统（Cursor、Codex、Gemini）——我们应明确链接到 agentskills.io 标准。

**要避免：** 纯星标排名。任何"awesome"话题的顶部结果通常被刷榜。对于知识市场，编辑策展胜过原始流行度。

## Skill Wiki v0.1.0 的综合建议

- **浏览优先**，不是搜索优先。语料库规模小（< 1,000 个原子）。
- **筛选侧边栏**，仅三个轴：类型、领域、边动词。
- **卡片布局**，含类型徽章、领域前缀、质量点、出边数、摘要行。悬停时显示复制安装按钮。
- **详情页**，带投影层级标签（Summary / Core / Full），加边面板和来源面板。
- **无星级，无下载量。** 用边出度和质量评分作为客观信号。
- **静态 HTML。** 无数据库，无鉴权。镜像 npm 安装发现流程：看见 → 复制 → 运行。
